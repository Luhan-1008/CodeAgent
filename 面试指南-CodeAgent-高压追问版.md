## 3. 高压问题清单（含答法模板）


## Q3：MCP 的实际价值是什么？

**回答模板：**

- 统一工具接入协议，减少“每接一个工具重写一套调用”的重复工作。

- 让 Agent 可以标准化调用搜索、抓取、执行等能力。

- 在多模型场景中，工具层抽象稳定，模型可替换，系统更易扩展。

**面试官追问“没有 MCP 也能做”时：**

- 承认“能做”，但成本会转移到维护：接口不统一、错误语义不一致、扩展成本高。

---

## Q4：你负责的 `LLMAgent` 链路，具体“核心”在哪？

**回答模板：**

- 核心在三个点：

  1. **消息与上下文组织**（输入、历史、记忆/RAG 的拼装）

  2. **工具调用闭环**（模型发起 tool call、执行结果回灌、继续推理）

  3. **异步执行与回调**（长任务不中断，可记录中间态）

- 我的贡献主要在稳定执行路径和可追踪性，而不是仅改 prompt。

### Q3/Q4 展开补充（含代码定位，可直接背）
#### Q3 补充：MCP 价值的工程化解释
- 面试时不要只说“统一协议”，要补一句：**它把工具接入复杂度从 Agent 主流程剥离到 MCP 层**，否则工具一多就会出现接口碎片化。
- 你们项目里，MCP 不是概念层，而是已经进入可运行链路：连接、列工具、调用、错误处理都在代码中落地。
- `code_agent/tools/mcp_client.py` 对应点：
  - `connect_to_server(...)`：支持 `stdio/sse/streamable_http/websocket`。
  - `connect(...)`：读取并连接 `mcpServers`，处理 `env/exclude/timeout`。
  - `get_tools(...)`：把 server 工具转成统一 `Tool` 结构。
  - `call_tool(...)`：统一返回工具调用结果/错误文本。
- `code_agent/tools/tool_manager.py` 对应点：
  - `connect(...)`：统一接 MCP 工具+本地工具。
  - `reindex_tool(...)`：做 `server---tool_name` 命名空间，避免重名冲突。
  - `single_call_tool(...)`：统一调用入口，含超时与异常兜底。
#### Q4 补充：LLMAgent 链路“核心”的可追问答案
- 推荐你按“执行时序”讲：`run_loop(...)` 初始化 → `step(...)` 单轮推理与工具调用 → `save_history/save_memory` 持久化。
- 这样讲比“我做了异步和工具调用”更像真实做过，因为能对上函数与执行顺序。
- `code_agent/agent/llm_agent.py` 对应点：
  - `run_loop(...)`：准备 LLM/Tools/Memory/RAG，进入多轮循环。
  - `step(...)`：memory condense → `llm.generate(...)` → tool call 分支 → 停止判定。
  - `handle_new_response(...)`：处理并记录 `tool_calls`。
  - `do_rag(...) / load_memory(...) / condense_memory(...)`：上下文增强挂载。
  - `read_history(...) / save_history(...)`：断点恢复与历史持久化。

## Q5：RAG 你们怎么做，怎么证明有价值？

**回答模板：**

- 不是把所有文档都塞进上下文，而是先检索相关片段再注入。

- 通过文档研究节点先产出结构化结论，再传给后续代码阶段。

- 价值体现在：减少幻觉、降低无关 token、提升答案一致性。

**追问“调参怎么做”：**

- 从 chunk 大小、重叠比例、检索 top-k、重排策略入手，按任务做权衡。

---

## Q6：自动修复会不会反复改坏代码？

**回答模板：**

- 修复阶段基于“最新错误日志”而不是盲改。

- 每轮修复后重新执行构建/测试，验证是否朝目标收敛。

- 设置最大迭代轮次与失败退出策略，避免无限循环。

---

## Q7：多模型适配有什么难点？

**回答模板：**

- 难点不在“API能调通”，而在能力差异：函数调用格式、上下文窗口、错误语义、速率限制都不同。

- 适配层要做统一抽象，同时保留模型特性（例如工具调用能力差异）。

- 目标是“业务流程一致，底层模型可替换”。

---

## Q8：你们如何做质量保障？

**回答模板：**

- 分为三层：

  1. **配置与流程可加载**（workflow 配置可校验）

  2. **关键路径测试**（核心节点/工具调用/执行链路）

  3. **端到端样例**（如 deep research 工作流）

- 高层目标是保证“从配置到执行”可验证，而不是只测单函数。

## Q9：项目最大技术风险是什么？

**推荐回答：**

- 外部依赖不稳定（工具服务、模型服务）会放大链路失败率。

- 缓解方式：超时控制、错误分类、重试/降级、观测日志。

---

## Q10：如果你来做下一版，优先级怎么排？

**回答模板：**

1. 先补评测体系（成功率、修复轮次、时延、token 成本）

2. 再做失败恢复（节点级重试、回滚、备选模型）

3. 最后做体验优化（更强可视化、调试与报告）

这样答能体现你有工程优先级思维。

## 6. 常见失分点（高压面最致命）

1. **把“参与”讲成“主导全部”**：一追问就露馅。

2. **乱报数字**：没有严格评测就不要给精确提升百分比。

3. **只讲概念不讲机制**：比如只说“用了 RAG”，却说不清检索如何进入流程。

4. **不承认边界**：正确做法是“讲清做了什么 + 没做什么 + 为什么”。

## 8. 基于源码的深挖补充（和你职责强绑定）

这一节用于应对“继续往下挖”的面试官，内容都能在代码里落到实现。

### 8.1 LLMAgent 执行链路可讲的细节

- Agent 初始化时会处理 `mcp_server_file` 与 `mcp_config` 合并。

- 支持 callback 生命周期（`on_task_begin/on_tool_call/after_tool_call/on_task_end`）。

- 有并行工具调用入口（`parallel_tool_call`），适合解释异步执行流。

**高压追问示例：**“你说执行链路，具体链路点是什么？”  

**回答建议：**“消息组织、工具调用回灌、回调钩子和异步并发，是我重点关注的 4 个稳定性抓手。”

### 8.2 ToolManager 的稳定性机制

- 并发限制：`asyncio.Semaphore(MAX_CONCURRENT_TOOLS)`。

- 超时保护：`asyncio.wait_for(..., timeout=tool_call_timeout)`。

- 失败兜底：返回可读错误文本，避免流程 silent fail。

**高压追问示例：**“工具失败后会怎样？”  

**回答建议：**“不会直接卡死，超时/异常会转换为可消费的错误信息回到模型侧继续决策。”

### 8.3 MCP 客户端的工程可扩展点

`MCPClient` 同时支持 `stdio/sse/streamable_http/websocket`，说明工具服务部署形态是解耦的。

**高压追问示例：**“为什么你说兼容性提升？”  

**回答建议：**“不是只兼容模型 API，也兼容不同工具服务传输协议，降低接入成本。”

### 8.4 Skill 主线如何避免‘只会背概念’

`SkillLoader` 的关键是：

- 识别 skill 目录（含 `SKILL.md`）

- 解析 skill schema

- 做 schema 校验

- 用 `skill_id@version` 做唯一键管理

这可以直接对应你简历里的“加载-解析-检索/管理”职责。

### 8.5 RAG 主线可讲的参数与模式

`LlamaIndexRAG` 可讲 4 个参数：

- `chunk_size`

- `chunk_overlap`

- `similarity_top_k`

- `score_threshold`

同时支持 `retrieve_only` 模式（只检索不走生成），适合对比不同成本/质量策略。

### 8.6 Memory 主线可讲的并发一致性

`Mem0Memory` 有 `asyncio.Lock`，并通过 `SharedMemoryManager` 基于用户/模型配置复用实例。

**高压追问示例：**“多任务并发下记忆会不会乱？”  

**回答建议：**“我们用锁保护共享记忆访问，并用配置键隔离不同用户/模型实例。”

### 8.7 Workflow + E2E 的证据链

- `projects/deep_code_research/workflow.yaml` 定义 6 节点 DAG 链路。

- `tests/test_deep_code_research_workflow.py` 覆盖：

  - workflow 加载

  - DagWorkflow 构建

## 10. 压力场景题（新增：函数级追问）

### 10.1 追问：工具调用超时你怎么处理？

答题结构：

1. 在 `ToolManager.single_call_tool` 里用 `asyncio.wait_for` 控制单次超时；

2. 超时不抛崩主流程，而是返回结构化错误文本给 Agent；

3. Agent 根据错误决定重试、降级或继续下一个步骤。

### 10.2 追问：DAG 为什么比链式更适合你们？

答题结构：

1. `DagWorkflow` 支持一个节点多个 `next`，更适合分支任务；

2. 用拓扑排序保证执行依赖正确；

3. 后续扩展并行分支时不用重写核心执行器。

### 10.3 追问：你如何证明自己做过 Skill 主线？

答题结构：

1. 先说输入类型（单目录、多目录、SkillSchema 对象）；

2. 再说目录识别与 schema 解析校验；

3. 最后说唯一键管理和 reload 逻辑。
